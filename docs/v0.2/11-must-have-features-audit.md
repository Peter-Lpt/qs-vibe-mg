# 必做功能总审计

> 审核目标：`docs/v0.2/10-must-have-features-technical-plan.md`

## 结论

通过，但必须按分层顺序推进。

## 审计摘要

这份总方案抓住了 backlog 里的核心矛盾：

- 不是缺页面，而是缺统一语义。
- 不是缺按钮，而是缺可信来源与动作边界。
- 不是缺更多功能，而是缺可控的优先级。

整体上，这份方案的划分是合理的，尤其是把“来源 / Agent / 同步安全”放到 UI 壳之前，这是对的。

## 子 agent 审核补充

子 agent 的审计意见和本地审计结果基本一致，且补强了几个关键约束：

- 未发现明确遗漏的 must-add 项。
- 优先级顺序总体合理，先基础能力、后壳层是对的。
- `统一动作模型` 有过早抽象风险，应该先做跨页共享契约。
- `Skill` 更新必须区分“可自动更新 / 仅 best-effort / 未知”，缺 provenance 时不得默认自动更新。
- 同步安全必须强制 dry-run，fallback 和失败原因都要可区分。
- 删除语义必须继续拆清，不能用一个入口覆盖所有对象。

## 审计结果

### 1. Skill 安装 / 更新 / 来源边界

通过。

原因：

- 这是后续自动更新、回放命令、来源可信度的底座。
- 没有这层，后面所有来源更新都会退化成猜。

风险：

- 来源记录如果写得太轻，会被误解成完全可靠。
- 如果 provenance 没有落实到更新分支，仍然会被当成装饰字段。

建议：

- 先把 provenance 定义清楚，再谈更新能力。

### 2. 来源识别与信任边界

通过。

原因：

- `明确记录 / 启发式 / 未知` 这个分层足够稳。
- 适合历史数据逐步接管，不会破坏现有内容。

风险：

- UI 文案如果太积极，用户还是会把推断当确定。

建议：

- 所有推断必须保留来源依据。

### 3. App 壳与整体视觉统一

通过，但放在第二梯队更合适。

原因：

- 这确实影响第一印象。
- 但如果先做它，底层语义问题还会继续泄露到体验里。

风险：

- 只做壳层，不碰模型，会出现“看起来更完整，实际上更混乱”。

建议：

- 和设置入口一起做，集中收口。

### 4. Agent 路径配置增强

通过。

原因：

- 这是本地多 agent 管理里非常实际的痛点。

风险：

- `skills_dir` 和 `detect_dir` 如果不分开，仍会误导用户。

建议：

- 用“检测目录”和“实际 skills 目录”双字段表达。

### 5. `AgentConfig` 能力模型完善

通过。

原因：

- `additional_scan_dirs` 和项目扫描模式，是避免硬编码的关键。

风险：

- 能力字段越加越多，配置会膨胀。

建议：

- 先加最小必要字段，不要一次性抽象成复杂平台模型。

### 6. 同步安全与 Windows 兜底

通过。

原因：

- 对桌面端本地文件管理来说，这是高频风险区。

风险：

- fallback 行为如果不提示，用户会误判操作结果。
- 如果失败原因不能区分，恢复路径会很难设计。

建议：

- dry-run、错误原因、恢复路径都要写清楚。

### 7. 统一动作模型

通过。

原因：

- 这能明显减少当前 list/tree/detail/batch 的重复逻辑。

风险：

- 如果只抽出 registry，不把所有视图接进去，收益会打折。
- 如果抽象过早，会拖慢前面的基础能力落地。

建议：

- 统一动作模型必须是跨页面收口，而不是局部封装。
- 先做共享契约，再逐步收敛到 registry。

### 8. 删除语义拆分

通过。

原因：

- 这是安全边界问题，不是 UI 文案问题。

风险：

- 如果继续一个“删除”按钮管所有对象，未来一定会出错。

建议：

- 删除语义必须先拆清，再考虑交互简洁性。

### 9. 问题修复入口增强

通过。

原因：

- 这是把“看问题”推进到“解决问题”的关键一步。

风险：

- 如果问题分组过去了，但策略上下文没过去，入口还是半成品。

建议：

- 问题分组和批量策略要一起传递。

## 需要特别注意的地方

1. 这不是九个独立项目，而是一套相互依赖的治理工程。
2. 必须先做来源、Agent、同步安全，再做壳层。
3. 删除语义和问题修复不要拖到最后，否则安全风险会持续存在。
4. App 壳改造适合作为收口项，不适合作为第一刀。

## 最终优先级

1. Skill 来源边界
2. 来源识别与信任边界
3. Agent 路径配置增强
4. `AgentConfig` 能力模型完善
5. 同步安全与 Windows 兜底
6. 统一动作模型
7. 删除语义拆分
8. 问题修复入口增强
9. App 壳与整体视觉统一

## 结论

这份总方案可以作为后续开发的统一前置文档。

它已经把必须加的功能全部覆盖到位，且把前置依赖和风险边界也说清了。子 agent 的审计也没有推翻这个判断，只是把 provenance、dry-run、动作抽象边界这几处约束压得更实了。下一步只要你一句“开始开发”，就可以按这里的优先级顺序逐项落地。
