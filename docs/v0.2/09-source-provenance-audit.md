# 来源溯源审计

> 审核目标：`docs/v0.2/08-source-provenance-technical-plan.md`

## 结论

通过

## 审计结果

- 方案范围收得住，不会一下子把 scan 流程搞乱。
- 来源记录是按 source 存的，和现有 `SkillSource` 模型一致。
- 保留旧数据的启发式兜底是对的，不会影响历史安装。
- 方案没有提前跳到自动更新逻辑，边界是稳的。

## 必须继续约束的点

1. `provider` 必须保持可空。
2. 启发式推断不能覆盖明确记录。
3. 前端在没有记录时必须继续显示“不确定”。
4. 如果来源记录写入失败，安装流程要明确报错。

## 风险

- 如果置信度文案太积极，用户还是可能把它当成确定事实。
- 如果解析过于宽松，坏记录会伪装成权威来源。
- 后续做更新路由时，不能把这份记录直接当作绝对真相，仍要校验。

## 实施检查

- Rust 里新增来源类型和读写 helper。
- 安装时持久化来源记录。
- 扫描时加载来源记录。
- 前端补一个明确的“已记录 / 推断 / 未知”展示。

## 结论

这份技术方案可以继续往下做，没有阻塞问题。
