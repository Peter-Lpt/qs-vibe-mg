# 筛选系统重构方案（初版审计稿）

## 1. 原始需求摘要

当前管理页筛选区域经历多轮局部修改后，出现以下系统性问题：

- 筛选入口、排序、Agent 范围、问题修复和批量选择混在同一区域，职责不清。
- 悬浮/粘滞布局会遮挡内容，并与应用顶部导航、工作台粘滞表头、底部批量栏产生层级冲突。
- 清除筛选入口不稳定，用户难以判断会清除搜索、状态、Agent 还是排序。
- 多个状态筛选采用全部 AND，可能出现“已链接 + 未链接”等逻辑上不可理解或结果恒为空的组合。
- 搜索无结果与系统内没有 Skill 使用相同空状态，反馈和下一步操作错误。
- Skill 数量多或 Agent 数量多时，上下滚动、横向滚动和筛选可达性不足。
- 全选、筛选结果和批量操作的作用域需要严格一致，避免隐藏项被误操作。

本次目标不是继续修补现有工具栏，而是重构筛选状态模型、信息架构、组件边界和滚动规则。

## 2. 初始判断

### 任务类型

- `refactor_or_optimization`：筛选状态、组件职责和滚动容器重构。
- `incremental_requirement`：必须兼容搜索、排序、Agent 筛选、状态筛选、问题修复、批量选择等现有能力。
- `bug_fix_plan`：解决遮挡、清除不可达、无结果反馈错误、互斥状态组合等问题。

### 必须重构

- 筛选状态模型。
- 筛选区信息架构。
- 筛选与问题修复、批量操作的职责边界。
- 工作台纵向与横向滚动策略。
- 空结果、零 Agent、多 Agent、大数据量等边界状态。

### 可以延后

- 保存自定义筛选方案。
- 跨启动恢复筛选条件。
- 高级表达式筛选。

### 不建议继续

- 继续在当前 `ManageTab.vue` 内增加布尔状态和模板分支。
- 使用负 margin、毛玻璃和高 z-index 模拟悬浮筛选条。
- 将问题修复卡片继续放在筛选表单内部。
- 允许互斥状态通过多选 Chip 任意组合。

## 3. 现有功能盘点

### 当前筛选能力

1. 搜索：名称、描述、ID，本地搜索，300ms 延迟。
2. 状态：冲突、断链、需要同步、未链接、已链接、未入库、仅库中、重复。
3. Agent 范围：多选 Agent，支持包含或排除。
4. 排序：状态优先、名称、来源数。
5. 计数：状态 Chip 显示数量。
6. 清除：清除搜索、状态、Agent 条件。
7. 全选：选择当前筛选结果。
8. 问题修复：按问题组选择 Skill 并进入批量操作。

### 当前耦合点

- `ManageTab.vue` 同时负责筛选状态、筛选算法、计数算法、筛选 UI、选择状态、批量面板和滚动布局。
- `IssueRepairPanel.vue` 既像问题摘要，又能触发批量选择，视觉上被放入筛选区后容易被理解为筛选条件。
- `SkillWorkbench.vue` 有内部纵向/横向滚动，页面本身也可纵向滚动。
- 页面底部批量栏使用 `fixed`，可能覆盖工作台横向滚动条和最后一行。

## 4. 最终目标模型

### 4.1 筛选状态结构

建议建立独立 `useManageFilters` composable：

```ts
interface ManageFilterState {
  query: string;
  relation: "all" | "needs_sync" | "linked" | "unlinked";
  issues: Set<"conflict" | "dangling" | "duplicate">;
  sourceScope: Set<"missing_library" | "library_only" | "agent_only" | "project_only">;
  agentIds: Set<string>;
  agentMatch: "any" | "all" | "exclude";
  sort: "status" | "name" | "sources";
}
```

逻辑规则：

- 搜索、关系状态、问题类型、来源范围、Agent 范围之间使用 AND。
- 同一“问题类型”组内使用 OR。
- 同一“来源范围”组内使用 OR。
- 关系状态是互斥单选，禁止同时出现“已链接”和“未链接”。
- Agent 默认 `any`；高级模式可切换 `all` 或 `exclude`。
- 排序不计入“已启用筛选数”，清除筛选默认保留排序。

### 4.2 筛选与动作解耦

- 筛选区只回答：“当前显示哪些 Skill”。
- 问题修复区只回答：“对哪些问题执行什么动作”。
- 批量选择只回答：“当前可见结果中选中了哪些 Skill”。
- `IssueRepairPanel` 从筛选面板移出，改为工作台上方的独立“问题摘要/快捷任务”入口。
- 点击问题摘要可以应用对应筛选预设，但不应直接静默选中隐藏 Skill。

## 5. 信息架构与交互方案

### 5.1 默认结构

```text
┌─────────────────────────────────────────────────────────────┐
│ 搜索 Skill [×]             排序        筛选(3)   清除筛选   │
├─────────────────────────────────────────────────────────────┤
│ 已启用：需要同步 ×  Claude Code ×  冲突 ×                  │
├─────────────────────────────────────────────────────────────┤
│ [展开区域]                                                     │
│ 关系状态：全部 / 需要同步 / 已链接 / 未链接                  │
│ 问题类型：冲突  断链  重复                                   │
│ Agent 范围：[多选控件]    匹配：任一 / 全部 / 排除           │
│ 来源范围：未入库  仅库中  仅 Agent  仅 Project              │
└─────────────────────────────────────────────────────────────┘

┌ 问题摘要：冲突 3 | 断链 2 | 未入库 8 | 打开修复中心 ┐
┌ Skill Workbench（自身滚动容器）                           ┐
```

### 5.2 工具栏规则

- 不使用页面级 `fixed` 或负 margin 悬浮。
- 筛选卡片保持正常文档流，避免覆盖应用导航和工作台。
- 搜索框内部始终提供独立清除按钮，仅清除关键词。
- “清除筛选”始终占位；无条件时禁用，有条件时高亮。
- 已启用条件以可移除 Token 显示，用户可单独删除条件。
- “更多筛选”展开区域采用 inline disclosure，不使用会遮挡表格的 Popover。
- 展开和收起不改变搜索、排序和清除按钮的位置。

### 5.3 Agent 数量自适应

- 0 个 Agent：隐藏 Agent 范围筛选，显示“暂无可筛选 Agent”。
- 1–5 个 Agent：显示紧凑复选项或 Token。
- 6 个及以上 Agent：使用可搜索的多选列表，面板内部最多显示约 6 行并独立滚动。
- Agent 名称过长时单行省略，完整路径通过 title/辅助说明查看。
- Agent 被删除或变为未检测状态时，自动移除失效筛选 ID，并显示一次轻量提示。

## 6. 滚动与遮挡规范

### 6.1 纵向滚动

- 页面只负责容纳标题、筛选和问题摘要。
- Skill Workbench 使用独立的弹性高度容器：`min-height: 0; flex: 1; overflow-y: auto`。
- 不再使用固定 `max-height: 66dvh` 作为唯一规则；优先由父级 Flex 布局分配剩余高度。
- 工作台表头在工作台内部 sticky，`top: 0`，不会与页面筛选栏竞争。
- 展开 Skill 详情时保持当前行可见，必要时仅滚动工作台容器，不滚动整页。

### 6.2 横向滚动

- Agent 列超宽时，工作台横向滚动条必须始终位于工作台可视边缘，而不是内容最底部。
- 对 6 个以上 Agent 提供顶部同步横向滚动轨道，底部保留原生滚动条，两者滚动位置同步。
- 固定第一列 Skill 信息和最后一列操作之前必须评估可视宽度；窄屏不同时固定两列，避免中间数据区被挤没。
- 横向滚动时 Agent 表头与单元格必须使用同一滚动容器，禁止分离错位。

### 6.3 遮挡与层级

建议层级：

- 页面内容：`z-index: auto`
- 工作台 sticky 表头：`10`
- 下拉/多选面板：`20`，仅在筛选卡片内部或 Teleport 后进行边界定位
- 底部批量操作栏：不再使用覆盖内容的全局 fixed；改为工作台容器内部 sticky footer，并为内容预留高度
- Modal：`50`

所有 sticky/fixed 元素必须满足：

- 有不透明背景。
- 有明确的占位或内容 padding。
- 不覆盖横向滚动条。
- 在 100%、125%、150% 缩放下不遮挡主要操作。

## 7. 全选与筛选作用域

- 全选只选择当前 `filteredSkills`。
- 筛选条件变化后，自动移除不可见选择，批量操作永远不包含隐藏项。
- 全选框支持未选、半选、全选三态。
- 结果为 0 时禁用全选。
- 数据刷新后移除已经不存在的 Skill ID。
- 问题修复预设在应用筛选后，不自动全选；需要用户再次确认全选或进入修复中心。
- 批量栏显示“已选 N / 当前结果 M”，而不是只显示全局已选数量。

## 8. 空状态和错误状态

### 搜索/筛选无结果

- 标题：“未找到匹配的 Skill”。
- 描述当前关键词和启用条件数量。
- 操作：“清除关键词”“清除全部筛选”。
- 不将“安装 Skill”作为第一操作，避免误导用户认为数据不存在。

### 系统中没有 Skill

- 标题：“暂无 Skill”。
- 操作：“安装 Skill”。
- 可提供技能目录说明。

### 加载和刷新

- 初次加载使用工作台骨架。
- 刷新时保留筛选条件；若条件引用的 Agent 已失效则清理该条件。
- 搜索结果计算必须基于同一份最新 Skill 数据，不保留过期 `searchResults` 快照。

### 错误

- 筛选计算本身不应产生后端错误。
- 数据加载错误保留筛选控件，但禁用依赖数据的计数和批量操作，提供重试。

## 9. 组件拆分建议

```text
ManageTab.vue
├─ ManageFilterBar.vue          搜索、排序、展开、清除
├─ ActiveFilterTokens.vue       当前条件与单项移除
├─ ManageFilterPanel.vue        关系/问题/来源/Agent 条件
├─ IssueSummaryBar.vue          问题摘要与修复入口
├─ SkillWorkbench.vue           展示和自身滚动
└─ SelectionActionBar.vue       当前结果范围内的批量操作

composables/
└─ useManageFilters.ts          状态、过滤、计数、清除、失效条件清理
```

组件不得各自重复计算过滤结果；唯一权威结果由 `useManageFilters` 输出：

- `filteredSkills`
- `activeTokens`
- `activeFilterCount`
- `facetCounts`
- `hasActiveFilters`
- `clearQuery()`
- `removeToken()`
- `clearFilters()`
- `normalizeAgainstAgents()`

## 10. 边界值清单

必须覆盖：

- Skill：0、1、2、100、1000。
- Agent：0、1、5、6、20。
- 搜索：空字符串、纯空格、中英文、大小写、特殊符号、超长文本。
- 状态：零结果、互斥状态、同时多个问题类型。
- Agent：不存在 ID、未检测、运行中删除、超长名称、相同显示名称。
- 选择：筛选前已选、筛选后隐藏、全选后刷新、结果归零。
- 屏幕：1024×640、1366×768、1920×1080；Windows 缩放 100%、125%、150%。
- 语言：中文、英文、繁体；按钮文本增长后不得挤压搜索框至不可用。
- 滚动：纵向长列表、横向 20 Agent、展开详情、批量栏显示、Modal 打开。
- 键盘：Ctrl/Cmd+K、Esc、Tab、Enter、Space，焦点不得落入隐藏区域。

## 11. 实施优先级

### P0：状态和职责正确性

1. 建立 `useManageFilters` 和明确的组合语义。
2. 将搜索改为直接基于最新 Skills 的 computed，去掉延迟快照状态。
3. 将问题修复从筛选区移出。
4. 明确清除、单项移除、全选和批量作用域。

### P1：规范化交互与滚动

1. 新建筛选栏、条件 Token 和 inline 展开面板。
2. 工作台改为剩余高度滚动容器。
3. 批量栏改为工作台内部 sticky footer。
4. 多 Agent 顶部/底部横向滚动同步。
5. 完整空状态和加载错误状态。

### P2：可访问性和性能

1. 键盘导航、ARIA 和焦点恢复。
2. 大列表计算与渲染性能测试，必要时再引入虚拟列表。
3. 保存筛选预设和持久化排序。

## 12. 验收标准

- 不存在逻辑互斥但可同时激活的关系状态。
- 清除关键词、清除单项条件、清除全部筛选都可在一个操作内完成且语义不同。
- 任何滚动位置都不会出现筛选栏、表头、批量栏和横向滚动条相互遮挡。
- 0/1/20 Agent 和 0/1/1000 Skill 均有明确可用的布局策略。
- 搜索无结果不会显示“新手 Skill”或错误的安装引导。
- 全选和批量操作不包含当前筛选结果之外的 Skill。
- 过滤算法、计数算法和 UI 使用同一个状态模型，不出现显示数量与结果不一致。
- 三种语言和 125%/150% Windows 缩放下主要操作不换行遮挡。
- 前端类型检查、构建和新增的筛选单元测试通过。

## 14. 子 Agent 审计结果

本节为最终收敛结论；若与前文初版设想冲突，以本节和第 15 节为准。

### 14.1 代码架构审计结论

代码审计确认以下问题成立：

- `ManageTab.vue` 同时承担筛选、计数、排序、选择、刷新、问题修复和布局，必须拆出纯筛选核心。
- Store 的 `searchResults` 是延迟快照，Skills 刷新后可能保留过期搜索结果，必须移除其结果真源地位。
- 当前状态筛选全部使用 AND，确实允许恒空组合。
- 当前工作台顶部全选框缺少半选状态，与底部批量栏不一致。
- `IssueRepairPanel` 当前会直接选择 Skill、打开批量面板或展开详情，不是纯摘要。
- `skillsStore.issues` 与 `IssueRepairPanel` 基于 Skill 字段重新分类形成两个问题真源。
- 当前 `missing_lib` 筛选与问题修复面板中的同名分类谓词不同。
- `refreshSkills()` 不可等待，并存在连续刷新覆盖和旧错误状态残留风险。

### 14.2 交互规范审计结论

交互审计确认以下问题必须提升到 P0：

- 必须明确唯一纵向滚动所有者，不能继续页面滚动、工作台滚动和覆盖式批量栏并存。
- 批量栏不能使用覆盖内容的 fixed/sticky，应作为工作台滚动区之外的布局兄弟节点。
- 必须定义最低有效视口和低高度退化规则，否则“无 UI 遮挡”不可验收。
- Agent `all`、顶部双横向滚动条、固定首尾列均会显著增加首期复杂度，应后置。
- 基础键盘操作、焦点恢复和 Modal 滚动锁定属于核心可用性，不应放到 P2。
- 问题摘要与问题筛选必须明确动作名称，不能让用户猜测点击后是筛选还是修复。

### 14.3 接受的建议

- 接受：问题组内 OR、跨组 AND。
- 接受：首期 Agent 只支持 `any/exclude`，不实现 `all`。
- 接受：引入独立 `useManageSelection`，不让筛选 composable 管理选择生命周期。
- 接受：Facet 采用“应用其他组，忽略当前组”的动态计数合同。
- 接受：批量操作栏改为滚动区外的普通布局行。
- 接受：用户主动改变筛选或排序后，工作台回到顶部；后台刷新尽量保持 Skill 锚点。
- 接受：先建立 Vitest 和纯函数特征测试，再替换现有筛选实现。
- 接受：保留现有三种排序，增加稳定二级排序。

### 14.4 后置的建议

- 后置：Agent `all` 匹配。
- 后置：顶部同步横向滚动条；先验证单一二维滚动容器是否足够。
- 后置：固定第一列/最后一列。
- 后置：虚拟列表；先建立 1000 Skill 性能预算和测量数据。
- 后置：筛选预设、跨启动持久化。

### 14.5 否决的方案

- 否决：问题摘要点击后直接全选并打开批量面板。
- 否决：工作台内容上方覆盖式 sticky footer。
- 否决：使用 `title` 作为长名称唯一查看方式。
- 否决：首期同时引入筛选语义、组件大拆分、滚动重构和虚拟列表。
- 否决：把当前 `independent` 不加定义地直接改名为互斥的 `needs_sync`。

## 15. 收敛后的最终方案

### 15.1 首期筛选模型

首期只保留用户已有且能明确解释的能力：

```ts
interface ManageFilterState {
  query: string;
  statusPreset: "all" | "needs_attention" | "linked_any" | "unlinked_all";
  issues: Set<"conflict" | "dangling" | "duplicate">;
  libraryScope: Set<"missing_library" | "library_only">;
  agentIds: Set<string>;
  agentMatch: "any" | "exclude";
  sort: "status" | "name" | "sources";
}
```

这里使用“状态预设”而不是宣称它们是互斥的数据分类：

- `all`：不限制状态。
- `needs_attention`：存在冲突、断链或需要人工同步的真实副本。
- `linked_any`：至少存在一个指向 Library 的 Agent 软链接。
- `unlinked_all`：不存在任何 Agent 软链接。

用户每次只能选择一个状态预设，因此不会出现“已链接 + 未链接”的恒空组合；不同 Skill 状态本身仍允许重叠。

### 15.2 精确谓词合同

所有组件必须调用同一组纯函数，禁止自行重复解析 `Skill.sources`：

```ts
classifySkillSources(skill)
matchesStatusPreset(skill, preset)
matchesIssues(skill, selectedIssues)
matchesLibraryScope(skill, selectedScopes)
matchesAgentScope(skill, agentIds, mode)
filterSkills(skills, state)
computeFacetCounts(skills, state)
```

来源分类规则：

- 优先使用 `source.source_kind`。
- 旧数据回退：`from === "vibe-lib"` 为 Library，`from.startsWith("project:")` 为 Project，其余已知 Agent ID 为 Agent，剩余为 External。
- `missing_library`：不存在 Library 来源。
- `library_only`：存在 Library 来源，且不存在 Agent、Project 或 External 来源。
- `missing_library` 与 `library_only` 不重叠。
- `agent_only/project_only` 首期不进入筛选模型，保留为问题诊断分类，待统一谓词后再评估。

问题组合规则：

- `issues` 组内 OR：匹配任一选中的问题类型。
- 空集合不限制结果。
- 冲突、断链、重复可以同时属于一个 Skill。

Agent 规则：

- 空 Agent 集合不限制结果。
- `any`：至少存在一个选中 Agent 的来源。
- `exclude`：不存在任何选中 Agent 的来源。
- 0 Agent 时隐藏 Agent 操作控件并显示一句说明。
- 1 Agent 时不显示匹配模式切换。
- 只有 Agent 列表成功刷新后才清理失效 ID；请求失败或加载中不得清理。

### 15.3 Facet 计数合同

每一筛选组的候选数量按以下方式计算：

1. 应用搜索和其他筛选组。
2. 忽略当前组已经选择的条件。
3. 单独应用当前候选项。
4. 显示应用该候选项后的结果数量。

补充规则：

- 零计数候选项可以禁用，但必须保留可见，避免布局跳动。
- 已选但因其他条件变为零的选项仍可取消，不得禁用取消动作。
- 问题组已选择多个条件时，Token 汇总显示并集结果，不将单项数量误写为总结果。
- 问题摘要中的总数按唯一 Skill ID 去重，不使用组长度直接求和。

### 15.4 搜索和刷新合同

- 搜索结果直接 computed 于最新 `skills` 和标准化 query。
- 不再持久保存 `searchResults` 数组。
- 如需防抖，只防抖 query 值，不缓存结果集合。
- `name`、`description`、`id` 必须按可空字符串安全处理。
- `refreshSkills()` 返回可等待 Promise。
- 仅最新一次刷新可以提交结果；成功刷新清除旧错误。
- 刷新完成后按顺序执行：提交 Skills → 规范化筛选 → 清理不存在选择 → 清理不存在展开项。

### 15.5 选择合同

新增 `useManageSelection`：

```ts
selectedIds
selectedVisibleCount
allVisibleSelected
partiallyVisibleSelected
toggleOne()
toggleAllVisible()
clearSelection()
pruneInvisible()
pruneMissing()
```

- 选择永远限制在当前筛选结果。
- 主动改变筛选后立即清理隐藏选择。
- 清除筛选后不恢复此前被清理的选择。
- 工作台顶部和批量栏使用同一三态计算。
- 批量栏显示“已选 N / 当前结果 M”。

### 15.6 问题入口合同

问题摘要从筛选面板移出，首期只提供两类明确动作：

- `筛选冲突`、`筛选断链`：显式应用筛选，不自动选择。
- `打开修复中心`：进入独立修复流程。

不得再由摘要按钮直接执行以下组合行为：

- 全选一组 Skill。
- 自动打开批量面板。
- 自动展开第一条 Skill。

问题筛选计数和问题摘要必须共享同一分类函数；后端 `skillsStore.issues` 若保留，只作为修复诊断详情，不作为同名计数的第二真源。

### 15.7 最终滚动结构

当前 `AppLayout.vue` 的内容容器是页面级 `overflow-y-auto`，因此工作台独占剩余高度需要先调整完整高度链。

目标结构：

```text
AppLayout
└─ ManageTab（h-full min-h-0 flex column）
   ├─ PageHeader（普通布局，可压缩）
   ├─ FilterArea（普通布局，不悬浮）
   ├─ IssueSummary（普通布局）
   └─ WorkbenchShell（flex-1 min-h-0）
      ├─ WorkbenchHeading
      ├─ TableViewport（flex-1 min-h-0 overflow-auto）
      └─ SelectionActionBar（普通布局行，不覆盖内容）
```

规则：

- Manage 页正常状态只有 `TableViewport` 承担纵向和横向滚动。
- History 页继续使用页面级滚动；`AppLayout` 需要按 Tab 选择滚动策略，或允许 ManageTab 接管高度。
- 表头 sticky 的滚动祖先就是 `TableViewport`。
- 横向滚动条位于 `TableViewport` 可视底边，不需要滚动到内容最后一行。
- 批量栏出现时通过 Flex 占用真实高度，自动缩小表格视口，不覆盖最后一行和横向滚动条。
- 筛选展开后，如果表格剩余高度低于 240px，筛选展开区自身限制高度并内部滚动，不继续压缩工作台。
- 页面不得因 Agent 列产生横向滚动。
- 首期不实现顶部第二横向滚动条；实测不满足后再进入 P2。

### 15.8 响应式退化顺序

空间不足时按以下顺序退化：

1. 排序按钮只显示当前值，不显示“排序”前缀。
2. 次要按钮改为图标 + Tooltip，但“清除筛选”保留可识别文字。
3. Active Token 最多显示两行，超出显示“另外 N 项”。
4. 排序进入“更多”菜单。
5. 工具栏换成两行，搜索框独占第一行。

约束：

- 搜索框最小可用宽度 220px。
- 0/1 Agent 不显示无意义控件。
- 英文和繁体文本增长不能造成页面横向滚动。
- 长名称使用可聚焦 Tooltip 或详情，不只依赖 `title`。
- 最低目标有效视口为 800×480 CSS px；低于该值允许筛选面板内部滚动，但不允许主要操作被完全遮挡。

### 15.9 键盘和焦点规则

首期必须完成：

- Ctrl/Cmd+K 聚焦搜索，但 Modal 打开或焦点位于可编辑控件时不抢占。
- Esc 按优先级关闭 Modal/下拉，再收起高级筛选，不清空查询。
- Token 删除后聚焦相邻 Token；没有相邻项时聚焦“更多筛选”。
- 清除关键词后焦点保留在搜索框。
- 清除全部筛选后焦点返回清除按钮或搜索框，并将工作台滚动到顶部。
- Modal 打开时锁定背景滚动；关闭后恢复原滚动位置和触发按钮焦点。
- 隐藏面板中的控件不可继续获得 Tab 焦点。

## 17. 最终验收场景

除第 12 节外，必须执行以下高风险场景：

1. 1024×640 窗口、150% 缩放、英文、展开高级筛选：工作台仍至少保留 240px 可操作高度，所有控件可达。
2. 20 Agent、1000 Skill，滚动到最右和最下：表头对齐，最后一行、最后一列和横向滚动条可操作。
3. 已选 5 项后筛选只剩 2 项：选择立即变为 2，两个全选框三态一致，批量目标只包含 2 项。
4. 显示批量栏并展开最后一行详情：详情和横向滚动条不被遮挡。
5. 0/1/20 Agent：控件分别按隐藏、简化、多选搜索策略展示。
6. 0 Skill 与筛选结果 0：分别显示安装引导与清除筛选引导。
7. 删除已筛选 Agent：仅在刷新成功后移除条件，提示一次，焦点和结果稳定。
8. 连续两次刷新后只接受最新结果，成功重试清除旧错误。
9. 一个 Skill 同时冲突、断链、重复：问题 OR、摘要去重和 Token 结果一致。
10. 使用 Tab、Shift+Tab、Enter、Space、Esc 完成整个筛选路径，焦点始终可见且不进入隐藏区域。
