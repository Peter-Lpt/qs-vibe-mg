# 必做功能总技术方案

> 来源：`docs/v0.2/07-feature-backlog.md`
>
> 目标：把 backlog 中所有“必须加”的功能收敛成一份可执行的总方案，先做设计和审计，不进入开发。

## 原始需求摘要

这次 backlog 的重点不是再找新点子，而是把当前产品的关键边界补齐。必须优先推进的内容包括：

1. `Skill` 安装 / 更新 / 来源边界
2. 来源识别与信任边界
3. App 壳与整体视觉统一
4. Agent 路径配置增强
5. `AgentConfig` 能力模型完善
6. 同步安全与 Windows 兜底
7. 统一动作模型
8. 删除语义拆分
9. 问题修复入口增强

## 初步判断

这些需求可以分成三层：

- 基础模型层：来源、Agent、路径、同步安全
- 交互模型层：统一动作、删除语义、问题修复
- 产品壳层：整体视觉、设置入口、版本信息

其中，基础模型层必须先定，否则后面的 UI 和批量动作会继续建立在不稳定语义上。

## 需求分组

### A. Skill 安装 / 更新 / 来源边界

#### 必须加

- 安装来源记录
- 可回放的安装 / 更新命令
- 版本、commit、最后检查时间、信任等级
- 更新时路由回正确来源
- 明确可识别来源与 best-effort 来源
- 先有来源元数据，再谈自动更新

#### 可延后

- Marketplace / plugin 深度解析
- 从 README、`.git/config`、额外元数据里推断来源

#### 设计要点

1. 为每个 `SkillSource` 增加来源记录字段。
2. 安装入库时写入 `.vibe-origin.json`。
3. 扫描时读取来源文件，缺失时才做推断。
4. UI 明确区分“已记录”“推断”“未知”。

#### 缺失约束

- 不能把没有 provenance 的目录当成自动可更新源。
- 不能把推断结果显示成确定事实。
- 更新状态必须分成“可自动更新 / 仅 best-effort / 未知”三类。
- 没有 provenance 时，默认不得进入自动更新分支。

### B. 来源识别与信任边界

#### 必须加

- 明确哪些来源可识别
- 明确哪些只做 best-effort
- 不要默认自动更新

#### 设计要点

1. 来源可信度分三级：明确记录 / 启发式 / 未知。
2. 任何更新都必须先验证来源类型和记录完整性。
3. 对于历史数据，只允许回填推断，不允许伪造确定来源。

#### 缺失约束

- provenance 不完整时必须保守显示。
- 自动更新逻辑必须显式要求来源记录存在。

### C. App 壳与整体视觉统一

#### 必须加

- 左上角品牌区重做
- 外层边框重复显示问题重构
- 按钮尺寸、图标大小、间距统一
- 设置入口重构
- 版本号、更新检查、版本升级放入设置

#### 设计要点

1. 先统一壳层，再细化局部组件。
2. 把版本相关信息从主视图中挪到设置页。
3. 全局按钮、卡片、图标大小采用统一 token。
4. Win / Mac 差异仅保留必要项。

#### 缺失约束

- 不能只修单个按钮或局部边框。
- 不能继续让版本信息占主内容区。

### D. Agent 路径配置增强

#### 必须加

- `detect_dir` 可编辑
- 内置 agent 的 `skills_dir` 可覆盖
- 路径校验提示

#### 设计要点

1. 区分“检测安装目录”和“实际 skills 目录”。
2. 允许用户覆盖内置 agent 的实际读写路径。
3. 路径校验要提示“目录存在”与“工具实际读取”不是一回事。

#### 缺失约束

- 不能只验证 skills 目录存在。
- 不能把公共目录误当成真实 agent 安装目录。

### E. `AgentConfig` 能力模型完善

#### 必须加

- `additional_scan_dirs`
- 可配置的项目扫描模式
- 不把所有项目约定永久写死在代码里

#### 设计要点

1. `AgentConfig` 增加能力描述字段，而不是只保留单一路径。
2. 项目扫描模式从后端默认值演进为可配置。
3. 保留向后兼容迁移逻辑。

#### 缺失约束

- 不能把项目规则硬编码成唯一真理。
- 不能用一个扁平字段承载所有来源语义。

### F. 同步安全与 Windows 兜底

#### 必须加

- symlink / junction / copy 失败提示
- 权限说明与恢复路径
- 默认关闭破坏性项目操作

#### 设计要点

1. 所有破坏性同步都要先做 dry-run。
2. Windows 下失败时要明确说明是权限、策略还是路径问题。
3. 项目目录默认只读，写回必须显式确认。
4. fallback 行为不能静默发生，必须显式提示。

#### 缺失约束

- 不能默认在 project source 上做破坏性写入。
- 不能让 copy fallback 默默发生而无说明。
- 失败原因必须可区分，不能只给一个笼统报错。

### G. 统一动作模型

#### 必须加

- 收敛 list / tree / detail / batch 的重复 action 逻辑
- 把公共动作行为收敛到同一套 action registry

#### 设计要点

1. 所有视图只负责展示，不各自实现动作判断。
2. 动作配置统一定义：可用性、危险级别、确认需求、文案。
3. 批量面板与详情页共享同一套动作判定。
4. 先做跨页共享契约，再逐步收敛到统一 registry，避免过早抽象。

#### 缺失约束

- 不能让不同页面各自维护一套动作 switch。
- 不能让同一语义在不同视图里显示不同文案。
- 不能为了“统一”先把动作系统抽象得过重。

### H. 删除语义拆分

#### 必须加

- 区分 library skill、agent link、断链清理、真实副本删除
- 默认保持 project 文件只读

#### 设计要点

1. 删除动作必须拆成显式入口。
2. project 删除默认禁用，只能走导入 / 预览 / 只读路径。
3. 清理断链与删除真实文件要分开确认。

#### 缺失约束

- 不能再用一个模糊“删除”覆盖所有对象。
- 不能把 project 文件和 library 文件混成同一删除语义。
- 必须能区分 library skill / agent link / 断链 / 真实副本删除。

### I. 问题修复入口增强

#### 必须加

- 保留问题分组上下文
- 批量模式预设策略
- 避免只选中条目，却丢掉修复意图

#### 设计要点

1. 问题分组进入批量面板时，传递类型上下文。
2. 面板根据上下文预设策略和默认动作。
3. 保留“冲突 / 断链 / 未入库 / 未覆盖目标”等分组。

#### 缺失约束

- 不能只把问题条目带过去，不带修复策略。
- 不能让用户进了批量面板还要重新猜问题类型。

## 总优先级

### 第一批

1. Skill 来源边界
2. 来源识别与信任边界
3. Agent 路径配置增强
4. `AgentConfig` 模型完善

### 第二批

5. 同步安全与 Windows 兜底
6. 统一动作模型
7. 删除语义拆分

### 第三批

8. App 壳与整体视觉统一
9. 问题修复入口增强

## 结论

这组必做项不是互相独立的零件，而是一套完整的管理语义修复工程。

最稳的顺序是先补来源和 Agent 模型，再统一动作和删除语义，最后做壳层和修复入口。这能避免 UI 先改漂亮了，底层语义却还是散的。
