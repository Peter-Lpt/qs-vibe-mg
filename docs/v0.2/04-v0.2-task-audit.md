# v0.2 全量任务排查

> 日期：2026-07-16  
> 范围：`docs/v0.2/01-manage-experience-optimization.md`、当前管理页实现、子 agent 审核反馈、AgentMatrix 任务追踪  
> 状态口径：已完成 / 部分完成 / 未开始 / 延后

## 总体结论

v0.2 当前已经完成“低风险体验一致性”和“批量 dry-run 雏形”的主要工作，但还没有完全达到方案文档中所有阶段目标。

已完成的重点：

- 管理页已将搜索、排序、Agent 范围、低频筛选和问题修复合并为紧凑的同步控制台，技能列表回到首屏主流程。
- 列表与来源/目录视图都具备预览、打开目录、复制路径等共享入口。
- 列表 agent 状态点已经增加 tooltip 和图例。
- 批量操作已有 dry-run 明细与执行结果分类。
- 问题修复入口已经出现，并能按任务组选择 skill。
- Project source 已以只读来源纳入扫描与来源树。
- 关系矩阵已降级为只读高级诊断入口。
- 拖拽释放后已改为确认预览，不直接执行。

仍需继续推进的重点：

- 统一动作模型还只是前端 composable 级别，详情页和来源树仍保留不少业务 switch。
- 问题修复入口还没有携带“问题类型上下文”进入批量面板。
- Project source 当前依赖后端 `current_dir()`，后续需要配置化项目根路径。
- 来源识别仍是 UI 级推断，没有实现 `.git/config`、README、`.vibe-origin.json` 的可靠追踪。
- 删除语义仍未完整拆成“删除 Library / 删除 agent 链接 / 删除断链 / 禁止删除 project”四种显式入口。

## 建议下一轮任务顺序

1. 删除语义拆分：先把危险动作边界彻底做清楚。
2. 问题修复入口携带上下文：例如点击“断链”直接预设清理断链策略。
3. 完整统一动作模型：让列表、来源树、详情、批量都从同一 action registry 派生。
4. Project root 配置化：不要长期依赖后端 `current_dir()`。
5. 来源识别后端化：读取 `.git/config`、README、`.vibe-origin.json`，并把置信度放到数据模型。
6. 补测试：Rust 路径安全、Project source 扫描、Batch dry-run 组件行为。
